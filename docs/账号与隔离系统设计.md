# 账号系统与多租户隔离设计 (Key-Based)

本系统采用轻量级的 Token 鉴权模式，通过唯一的 **User Key** 实现用户身份识别、配置隔离及结果存取。

## 1. 核心概念

- **User Key**: 系统生成的随机复杂字符串（例如：`fzt_2a8b9c...`），是用户唯一的身份凭证。
- **配置绑定**: 每个 User Key 关联一个独立的 `OpenAI API Key` 或其他模型配置。
- **数据隔离**: 每个 User Key 拥有独立的存储空间，互不干扰。

## 2. 交互流程

### A. 账户注册 (Registration)
1. 用户访问注册页面/接口，提交所需的初始配置（如 OpenAI Key）。
2. 系统验证频率限制（3分钟内最多5次）。
3. 系统生成唯一的 `User Key`。
4. **关键点**：完整 Key 仅在注册成功的响应中显示一次。

### B. 身份验证 (Authentication)
1. 客户端在请求 API 时，需在 Header 中包含：`X-FZT-Key: <User-Key>`。
2. 后端校验 Key 的有效性：
   - 校验通过：加载对应用户的 `profile.json`。
   - 校验失败：返回 `401 Unauthorized`。

### C. 数据存储隔离 (Isolation)
所有用户数据按 Key 的哈希值（SHA256）进行分目录存储，且不同业务任务的运行记录实施物理子目录隔离，不再使用共享的 `latest_run.json` 指针文件，以彻底规避多服务并发时的状态覆盖问题。

目录结构规范如下：

```text
storage/accounts/
  └── {key_hash}/
      ├── profile.json                  # 用户的模型与全局配置
      └── runs/                         # 异步任务运行历史
            ├── diagnosis/              # [NEW] 门店经营诊断任务子目录
            │     └── {run_id}/         # 每次运行生成的唯一 ID 文件夹
            │           ├── session.json# 任务状态与精简结果
            │           ├── logs.json   # 任务日志
            │           └── workspace/  # 大 Agent 运行工作区
            │
            └── price_recommendation/   # [NEW] 最优价格推荐任务子目录
                  └── {run_id}/
                        ├── session.json
                        ├── logs.json
                        └── workspace/
```

### 任务寻址与最新状态定位

当系统启动、重启或需要查询某项业务最新的任务进度（Latest Run）时，定位算法为：
1. 直接扫描该业务服务对应的物理目录（如 `runs/diagnosis/` 或 `runs/price_recommendation/`）。
2. 将获取的子文件夹列表按文件夹名称（即 `run_id` 的时间戳，如 `20260526T133231_b5abab`）进行降序排序。
3. 从最新到最旧依次尝试读取其下的 `session.json`，第一个成功读取并解析的文件即为当前业务的最新任务。
4. 诊断分析服务会自动向后兼容老系统直接写在 `runs/` 根目录下的历史任务索引。

## 3. 安全与限制

- **注册频率限制**: 
  - 窗口期：180 秒。
  - 最大次数：5 次。
  - 超限返回：`429 Too Many Requests`。
- **掩码显示**: 
  - 除注册接口外，所有查询接口返回的 Key 均处理为 `fzt_a...xyz` 格式。

## 4. 接口预览 (API)

| 接口 | 方法 | 说明 |
| :--- | :--- | :--- |
| `/api/auth/register` | `POST` | 注册新 Key |
| `/api/auth/verify` | `POST` | 验证 Key 是否有效 |
| `/api/analyze` | `POST` | 带 Key 分析 (Header: X-FZT-Key) |
| `/api/status` | `GET` | 获取该 Key 对应的结果 |
