# DeepSeek API 上下文缓存命中指导文档

## 1. 核心结论

DeepSeek API 的缓存是**上下文硬盘缓存 / Context Caching**，并且**默认开启**，用户**无需改接口、无需手动创建 cache key**。每个请求都会触发缓存构建；后续请求如果和之前请求在**前缀**上重复，重复部分可从缓存读取，计入缓存命中。([DeepSeek API Docs][1])

重点：它命中的不是“任意重复文本”，而是**从第 0 个 token 开始的重复前缀**。中间某一段重复，不算命中。官方新闻页也明确说：只有两个请求的**前缀内容相同**才算重复，中间开始的重复不能命中。([DeepSeek API Docs][2])

---

## 2. 缓存怎么命中

DeepSeek 当前规则里，缓存命中的前提是对应前缀已经被**落盘**。并且每条缓存前缀是一个**独立完整单元**；后续请求必须**完整匹配这个缓存前缀单元**，才能命中。([DeepSeek API Docs][1])

官方列出的缓存前缀落盘时机有三种：([DeepSeek API Docs][1])

1. **请求结束位置落盘**
   每次请求的**用户输入结束位置**和**模型输出结束位置**会产生缓存前缀单元。

2. **公共前缀检测落盘**
   多次请求之间存在公共前缀时，系统会把这个公共前缀作为独立缓存前缀单元落盘。

3. **固定 token 间隔落盘**
   对长输入或长输出，系统会按一定 token 间隔截取缓存前缀单元，避免长前缀迟迟无法被缓存。

## 3. 官方示例抽象

### 能命中的情况

第一次请求：

```text
A + B
```

第二次请求：

```text
A + B + C
```

第二次可以完整匹配第一次落盘的 `A + B` 缓存前缀单元，所以 `A + B` 可以命中。([DeepSeek API Docs][1])

### 不是立刻命中的情况

第一次请求：

```text
A + B
```

第二次请求：

```text
A + C
```

第二次**不能**命中第一次的 `A + B`，因为 `A + C` 没有完整匹配 `A + B` 缓存前缀单元。但系统可能识别出公共前缀 `A` 并落盘；第三次请求 `A + D` 时，就可能命中 `A`。([DeepSeek API Docs][1])

---

## 4. Prompt 组织原则

为了提高命中率，请把内容按这个顺序组织：

```text
稳定、重复、长的公共内容
+
本次请求变化的内容
```

也就是：

```text
system 固定提示词
few-shot 示例
固定背景材料 / 固定上下文
固定格式要求
---
本次具体问题 / 本次变量
```

不要反过来写成：

```text
本次变量
+
固定长提示词
+
固定材料
```

因为 DeepSeek 只看**前缀重复**，后面重复不等于命中。

---

## 5. 最容易导致不命中的写法

避免在公共前缀前面或中间插入这些动态内容：

```text
当前时间
随机 UUID
请求 ID
用户临时输入
节点运行次数
临时文件路径
动态排序后的 JSON
```

例如：

```text
请求时间：2026-05-12 18:30
固定 system prompt
固定 few-shot
固定材料
本次问题
```

这种会让第 0 个 token 开始就变化，公共前缀被破坏。

更适合写成：

```text
固定 system prompt
固定 few-shot
固定材料
本次问题
请求时间：2026-05-12 18:30
```

## 6. 多轮对话的缓存

官方示例中，第二轮请求如果包含第一轮的完整消息前缀：

```json
[
  { "role": "system", "content": "..." },
  { "role": "user", "content": "第一轮问题" },
  { "role": "assistant", "content": "第一轮回答" },
  { "role": "user", "content": "第二轮问题" }
]
```

那么前面的历史上下文可以命中缓存。([DeepSeek API Docs][1])

所以，多轮对话场景天然适合缓存，因为每一轮通常都会携带前面完整历史。

---

## 7. 长文本问答的缓存

官方长文本示例是：

第一次：

```text
system 固定角色
<财报内容>
请总结关键信息
```

第二次：

```text
system 固定角色
<财报内容>
请分析盈利情况
```

第三次：

```text
system 固定角色
<财报内容>
请分析收入与支出占比
```

官方说明：前两次不一定命中；前两次完成后，系统会识别 `system + <财报内容>` 为公共前缀并落盘，第三次完整复用该前缀时可以命中。([DeepSeek API Docs][1])

## 8. 怎么看有没有命中

DeepSeek API 返回的 `usage` 中有两个字段：([DeepSeek API Docs][1])

```json
{
  "prompt_cache_hit_tokens": 12345,
  "prompt_cache_miss_tokens": 678
}
```

含义：

- `prompt_cache_hit_tokens`：输入中**命中缓存**的 token 数。
- `prompt_cache_miss_tokens`：输入中**未命中缓存**的 token 数。
- `prompt_tokens = prompt_cache_hit_tokens + prompt_cache_miss_tokens`。([DeepSeek API Docs][3])

Python 示例：

```python
usage = response.usage

hit = usage.prompt_cache_hit_tokens
miss = usage.prompt_cache_miss_tokens

hit_rate = hit / (hit + miss) if hit + miss else 0

print("cache hit tokens:", hit)
print("cache miss tokens:", miss)
print("cache hit rate:", hit_rate)
```

如果是**流式输出**，需要设置：

```json
"stream_options": {
  "include_usage": true
}
```

官方说明：这样会在 `[DONE]` 前额外传一个 chunk，其中 `usage` 字段显示整个请求的 token 使用统计。([DeepSeek API Docs][3])

## 9. 计费理解

DeepSeek 的价格页按 **每 1M tokens** 计费，费用规则是：

```text
费用 = token 数 × 单价
```

官方当前价格页显示，输入 token 分为：

- **cache hit 输入价**
- **cache miss 输入价**
- **输出价**

例如当前价格页中，`deepseek-v4-flash` 的 cache hit 输入价明显低于 cache miss 输入价；`deepseek-v4-pro` 也有单独的 cache hit 输入价。具体价格以官方价格页为准，因为官方说明产品价格可能变化。([DeepSeek API Docs][4])

成本公式：

```text
input_cost =
  prompt_cache_hit_tokens  × cache_hit_price
+ prompt_cache_miss_tokens × cache_miss_price

output_cost =
  completion_tokens × output_price
```

## 10. 缓存不会改变输出内容机制

官方说明：硬盘缓存只匹配用户输入的**前缀部分**；输出仍然通过推理计算得到，仍受 `temperature` 等参数影响。因此，使用缓存和不使用缓存的输出效果相同。([DeepSeek API Docs][1])

换句话说：

```text
缓存 = 省输入前缀计算成本 / 降低延迟
不是 = 固定返回旧答案
```

---

## 11. 限制与注意事项

官方明确说明：([DeepSeek API Docs][1])

- 缓存是**尽力而为**，不保证 100% 命中。
- 缓存构建耗时是**秒级**。
- 缓存不再使用后会自动清空，时间一般是**几个小时到几天**。

另外，`user_id` 可用于 KVCache 缓存隔离，以进行隐私管理。([DeepSeek API Docs][3])

## 12. 推荐检查清单

每个请求发出前检查：

```text
[ ] 大段固定内容是否放在最前面？
[ ] 本次变化内容是否放在最后面？
[ ] 公共前缀里有没有时间、随机数、请求 ID？
[ ] JSON / Markdown / 空格 / 换行是否稳定？
[ ] few-shot 示例是否固定顺序？
[ ] 多轮对话是否保留完整历史前缀？
[ ] 是否记录 prompt_cache_hit_tokens / prompt_cache_miss_tokens？
[ ] 流式输出是否设置 stream_options.include_usage = true？
```

一句话原则：**把会重复的大块内容稳定地放在输入最前面，把每次变化的东西全部后置。**
