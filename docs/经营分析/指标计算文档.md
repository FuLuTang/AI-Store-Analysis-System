# 指标计算文档

> 相关文档：[架构设计](./架构设计.md) | [analysis_params 规范](./analysis-params-spec.md)

---

## 1. 文档定位

本文档用于定义 custom workflow 的指标口径与输出约束。

它的作用是：

1. 告诉 Agent 常见经营字段应该如何理解。
2. 约束指标输出格式和状态枚举。
3. 提供常见指标的计算口径和 SQL 写法方向。
4. 减少 Agent 每次都重新发明字段名、指标名和报告口径。

执行时，Agent 读取本文件并结合实际表结构计算指标。

---

## 2. 产物约束

Agent 计算完成后，指标结果写入 `output/指标.json`。

每条指标至少包含：

```json
{
  "metric_id": "gross_margin_rate",
  "name": "毛利率",
  "value": 0.31,
  "unit": "%",
  "status": "attention",
  "reason": "本期毛利率低于近三个月常态水平",
  "evidence": [
    {
      "type": "sql",
      "detail": "SELECT SUM(\"毛利\") / NULLIF(SUM(\"营收\"), 0) AS gross_margin_rate FROM sales"
    }
  ]
}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `metric_id` | 是 | 稳定英文 ID，便于程序和前端识别 |
| `name` | 是 | 中文指标名 |
| `value` | 是 | 指标值，可为数值、文本或对象 |
| `unit` | 否 | `%`、`元`、`次`、`人` 等 |
| `status` | 是 | `pass` / `attention` / `warning` / `uncountable` |
| `reason` | 是 | 为什么是这个状态，必须落到数据事实上 |
| `evidence` | 否 | 支撑证据，保存 SQL、样例表或对比结果 |

---

## 3. 状态枚举

| 状态 | 含义 |
|------|------|
| `pass` | 表现正常或无明显风险 |
| `attention` | 需要关注，但不一定已经恶化 |
| `warning` | 风险较高，值得优先行动 |
| `uncountable` | 数据缺失、口径不足或无法可靠计算 |

使用规则：

- 缺少必要字段时，直接返回 `uncountable`
- 除零、空表、严重脏数据时，直接返回 `uncountable`
- 不要为了输出完整而编造数值
- `reason` 必须给出可核验原因，不允许只重复状态名

## 4. 标准语义字段

这些字段用于字段语义映射，不是数据库强制 schema。

### 4.1 时间字段

| 标准字段 | 含义 | 常见原字段 |
|----------|------|------------|
| `date` | 日期 | 日期、day、业务日期、created_at |
| `month` | 月份 | month、月份、账期 |
| `time_slot` | 时段 | 小时、时段、班次、meal_period |

### 4.2 经营结果字段

| 标准字段 | 含义 | 常见原字段 |
|----------|------|------------|
| `revenue` | 营收 | 营业额、销售额、GMV、turnover |

| `customer_count` | 来客数 | 客流、访客数、顾客数 |

### 4.3 商品与维度字段

| 标准字段 | 含义 | 常见原字段 |

## 5. 字段理解规则

Agent 做字段映射时，优先顺序应当是：

1. 看字段名本身
2. 看样例值
3. 看表之间关系
4. 看时间粒度和业务语境

例如：

- `金额` 不必然是 `revenue`，也可能是 `cost`、`discount_amount`、`inventory_amount`
- `数量` 不必然是销量，也可能是库存量、来客数、订单数
- `评分` 在经营场景通常更接近 `rating`，在人事场景可能是 `performance_score`

字段不确定时：

- 可以在 `reason` 或 `evidence` 中说明映射依据
- 如果确实无法可靠判断，对相关指标返回 `uncountable`

---

## 6. 指标设计原则

当前经营分析采用 Agent 执行模式。此处定义指标集合和口径约束，不要求每次全量计算。

Agent 应当：

1. 结合真实表结构，确认哪些字段可用。
2. 优先计算业务含义清楚、证据稳定的指标。
3. 对口径不足的指标直接放弃，不要硬算。
4. 每算出一个指标，都尽量留下 SQL 或结果表作为 `evidence`。

## 7. 常用经营指标

### 7.1 营收

- `metric_id`: `revenue_total`
- `name`: `总营收`
- 必要字段：`revenue`

SQL 示例：

```sql
SELECT SUM("营收") AS revenue_total
FROM sales
```

### 7.2 订单量

- `metric_id`: `order_count_total`
- `name`: `总订单量`
- 必要字段：`order_count`

### 7.3 客单价

- `metric_id`: `avg_order_value`
- `name`: `客单价`
- 必要字段：`revenue`, `order_count`

SQL 示例：

```sql
SELECT SUM("营收") / NULLIF(SUM("订单数"), 0) AS avg_order_value
FROM sales
```

### 7.4 毛利率

- `metric_id`: `gross_margin_rate`
- `name`: `毛利率`
- 必要字段：`gross_profit`, `revenue`

SQL 示例：

```sql
SELECT SUM("毛利") / NULLIF(SUM("营收"), 0) AS gross_margin_rate
FROM sales
```

### 7.5 会员销售占比

- `metric_id`: `member_revenue_share`
- `name`: `会员销售占比`
- 必要字段：`member_revenue`, `revenue`

### 7.6 折扣率

- `metric_id`: `discount_rate`
- `name`: `优惠占比`
- 必要字段：`discount_amount`, `revenue`

### 7.7 库存周转相关

- `metric_id`: `inventory_turnover_proxy`
- `name`: `库存周转代理指标`
- 必要字段：视数据而定，常见为 `inventory_qty` / `inventory_amount` / `revenue`

约束：

- 如果缺少期初期末库存、采购明细或销量明细，就不要伪造“标准库存周转率”
- 可以输出代理指标，但必须在 `name` 或 `reason` 中说明是代理口径

### 7.8 渠道集中度

- `metric_id`: `channel_concentration`
- `name`: `渠道集中度`
- 必要字段：`channel`, `revenue`

SQL 示例：

```sql
WITH channel_rev AS (
  SELECT "渠道", SUM("营收") AS revenue
  FROM sales
  GROUP BY 1
)
SELECT MAX(revenue) / NULLIF(SUM(revenue), 0) AS channel_concentration
FROM channel_rev
```

### 7.9 商品 / 品类 TOP 贡献

- `metric_id`: `top_product_contribution`
- `name`: `TOP商品贡献度`
- 必要字段：`product_name`, `revenue`

应附带明细表作为 `evidence`。

### 7.10 趋势指标

可按时间维度计算：

- 营收环比
- 订单环比
- 毛利率趋势
- 会员销售趋势

前提：

- 必须有明确时间字段
- 粒度至少能形成两个以上连续时间点

---

## 8. SQL 编写原则

1. 优先用实际列名，不强行改列名
2. DuckDB 标识符统一用双引号包裹
3. 分母统一加 `NULLIF(..., 0)`
4. 先聚合再算比率，避免行级平均导致口径错误
5. 时间趋势类指标先按时间聚合再比较
6. 输出给报告的值尽量保留业务上可解释的精度

示例：

```sql
SELECT
  "月份",
  SUM("营收") AS revenue
FROM sales
GROUP BY 1
ORDER BY 1
```

---

## 9. Evidence 写法

`evidence` 使用数组结构，每项必须带 `type`。

类型示例：

```json
[
  {"type": "sql", "detail": "SELECT ..."},
  {"type": "table", "detail": [{"channel": "美团", "revenue": 12345}]},
  {"type": "note", "detail": "缺少成本字段，因此未计算标准毛利率"}
]
```

要求：

- 至少保留一条 SQL 或结果表
- 如果指标不可算，把不可算原因也写进 `evidence`
- 不要只给结论，不给依据

---

## 10. 禁止项

Agent 不应当：

- 在字段明显不够时硬算指标
- 为了凑完整报告编造行业均值或历史值
- 把大文件全文读进上下文再硬算
- 忽略 `uncountable`，把缺数据问题包装成业务结论

---

## 11. 最小交付标准

一次合格的经营分析，至少应满足：

1. `output/指标.json` 可解析
2. 每条指标都有稳定 `metric_id`
3. `status` 只使用四种合法枚举
4. `reason` 不空泛，能落到数据
5. 至少关键指标有 `evidence`
6. `summary.md` 与 `summary_short.json` 的结论不互相冲突

优先保证可核验性与一致性，不以指标数量为目标。
